## 第一轮：独立审核发现

### SubAgent A（漏洞检测性能专家）

**5 个最严重问题：**

1. **阶段迭代预算与任务量严重失衡**：`max_iter_per_state` 默认 6，而 `VULN_DETECT` 阶段可为 12 个参数生成 48-60 个任务，多数检测器根本轮不到执行，漏洞覆盖率被硬截断。（`config/settings.py` 第 77 行、`machine.py` 第 296-322 行）

2. **SQLi 时间盲注仅覆盖 MySQL**：`vuln_patterns.py` 定义了 PostgreSQL/MSSQL 载荷，但 `sqli_detector.py` 只用 `time_payloads_mysql[0]`，其他 DB 族直接漏检。（`sqli_detector.py` 第 102 行）

3. **sqlmap 适配器"高成本低深度"**：`--flush-session` 禁用缓存、`--level=1 --risk=1` 限制检出能力、120s 子进程超时与批任务叠加成瓶颈。（`sqlmap_scan.py` 第 59-76 行）

4. **HTTP 层全局串行 + sleep 节流**：所有工具的 HTTP 请求串行执行且受 `request_delay` 节流，端到端扫描时间被放大数倍。（`_http.py` 第 83-84 行）

5. **SSRF 检测启发式过于宽松**：仅靠响应长度差分和关键词（如 `ec2`/`iam`）判定，高误报流量会占用后续阶段预算。（`ssrf_detector.py` 第 69-72 行）

**否决的设计：**
- 对所有参数做全笛卡尔积式任务规划，无先验排序和剪枝
- SSRF 宽松启发式直接标记为 finding

---

### SubAgent B（渗透测试自动化专家）

**5 个最严重问题：**

1. **迭代预算与任务量错配（同 A-1）**：每步只执行一个工具调用，6 次迭代无法完成长任务队列，阶段"伪完成"。（`orchestrator.py` 第 106-180 行）

2. **EXPLOIT 在禁用利用时行为等于空跑**：任务标为 `pending_approval` 后 MockBrain 直接 `advance`，既不阻塞也不审批。（`mock_brain.py` 第 24-30 行、`machine.py` 第 327-330 行）

3. **LLM 失败时静默降级为 Mock**：调用方仍以 LLM 模式展示，但决策已变为规则机，观测性不足。（`orchestrator.py` 第 91-96 行）

4. **ReAct 闭环不完整**：工具执行后同一步不二次征询模型，需等下次迭代才能喂回观测，与迭代预算叠加浪费严重。（`orchestrator.py` 第 170-180 行）

5. **REPORT 阶段未真正编排**：状态机中 REPORT 仅发 enter/exit 事件，LLM 从未参与报告生成。（`machine.py` 第 86-90 行）

**否决的设计：**
- `LLMError` 时自动切 MockBrain 且不打标
- `phase_tasks` 白名单约束但无纠错/重选编排

---

### SubAgent C（安全工程与利用链专家）

**5 个最严重问题：**

1. **Web API 未做目标域校验**：CLI 调用 `ensure_target_allowed`，但 `POST /api/scan` 没有，攻击面可扩展到内网。（`api.py` 第 132-149 行 vs `autopnex.py` 第 125-128 行）

2. **审批令牌允许 `target="*"` 通配**：可授权任意目标，属于策略层横向放大。（`policy.py` 第 78-80 行）

3. **工具层 URL 无二次校验**：`ensure_target_allowed` 仅在 CLI 入口调用一次，工具内部对任意传入 URL 直接发请求，LLM 指定非目标主机时无拦截。（`_http.py` 第 86-97 行）

4. **审批签名与 LLM 密钥耦合**：`_secret()` 用 DeepSeek API Key 的 SHA256，无密钥时回退固定串，审批边界不可靠。（`policy.py` 第 32-34 行）

5. **利用结果不更新 Finding 状态**：`ingester.py` 的 `sqli_exploit`/`finding_replay` 只往 `exploit_evidence` 追加 dict，不把 Finding 升为 `exploited`，证据链断裂。（`ingester.py` 第 220-246 行）

**否决的设计：**
- 用 LLM 密钥作 HMAC 主密钥
- 全局 `verify=False` + CORS `*` + 无鉴权扫描 API

---

### SubAgent D（UI 交互与可观测性专家）

**5 个最严重问题：**

1. **SSE 断线无重连**：`EventSource` 的 `onerror` 仅记录并关闭，无 `Last-Event-ID`、无重连逻辑。（`app.js` 第 207-210 行、`api.py` 第 193-208 行）

2. **finding 状态更新被忽略**：`mergeFindings` 用 key 去重，已存在时跳过不更新，`confirmed -> exploited` 等状态变更丢失。（`app.js` 第 472-481 行）

3. **telemetry 不完整**：`performance_snapshot` 的 `total_duration_ms` 等关键字段被丢弃。（`app.js` 第 338-344 行）

4. **工具输出双重截断**：编排器截 800 字符，前端再截 240 字符，无展开或下载路径。（`orchestrator.py` 第 145 行、`app.js` 第 294-296 行）

5. **报告模板对敏感内容缺少转义策略**：`payload`/`evidence` 直接输出，`jinja2` 关闭 `autoescape`，HTML 报告有 XSS 风险。（`report.md.j2` 第 82-88 行、`generator.py` 第 27-28 行）

**否决的设计：**
- SSE 用 `get_nowait` + `sleep(0.25)` 轮询
- 前端忽略丰富事件类型，用单一 `<pre>` 承载所有信息

## 第二轮：交叉否决

### 四方交叉否决焦点

| 被否决方 | 否决方 | 否决点 | 原因 |
|---------|--------|--------|------|
| A 的全笛卡尔积规划 | B | 迭代预算不够用 | 任务爆炸但执行窗口有限，LLM 模式下更严重 |
| B 的 LLM 静默降级 | C | 安全审计不可信 | 渗透报告无法区分"模型决策"还是"规则决策" |
| C 的审批通配 `*` | A | 性能测试被滥用 | 通配授权下外部工具可打任意目标，资源失控 |
| D 的 SSE 轮询 | A | 延迟与吞吐浪费 | 忙等事件循环时间，实时性与性能都受损 |
| A 的 SSRF 宽松启发 | C | 假阳性污染证据链 | 低置信 finding 进入 exploit 规划，浪费利用预算 |
| B 的 REPORT 空跑 | D | 用户以为有 LLM 报告 | prompts.py 有 REPORT 描述但从未调用，误导用户 |

### 否决后共识

经过交叉否决，四方一致同意以下 **不可接受的现状**：
1. 迭代预算 6 次 vs 50+ 任务的失衡（A+B 一致否决）
2. Web API 缺少目标域校验（C 发现，A+B+D 一致认为严重）
3. 审批令牌通配 + 签名与 LLM 密钥耦合（C 发现，全组一致否决）
4. SSE 无重连且 finding 状态不更新（D 发现，A+B 确认影响性能展示）
5. LLM 失败静默降级（B 发现，C 确认影响审计可信度）

## 会议摘要

### 会议主题
AutoPenX 项目从漏洞测试性能和自动化程度角度的严格审核

### 讨论过程

**第一轮（独立审核）**：
- A 发现检测预算失衡、SQLi 覆盖缺口、sqlmap 高成本低深度
- B 发现迭代瓶颈、LLM 静默降级、REPORT 空跑
- C 发现 Web API 无目标校验、审批通配、签名密钥耦合、证据链断裂
- D 发现 SSE 无重连、finding 状态丢失、工具输出截断

**第二轮（交叉否决）**：
- A 否决 D 的 SSE 轮询方案（性能浪费）
- B 否决 A 的全笛卡尔积规划（预算不够用）
- C 否决 B 的静默降级（审计不可信）和 A 的 SSRF 宽松启发（误报污染）
- D 否决 B 的 REPORT 空跑（误导用户）

**第三轮（共识收敛）**：
- 四方一致：预算失衡、目标校验缺失、审批通配是最紧急的三个问题
- 四方一致：12 条建议全部通过，按 P0/P1/P2 三级排序
- 四方一致：5 条架构原则作为后续开发的底线约束

### 最终结论

AutoPenX 的架构骨架（状态机 + LLM 编排 + 模块化工具 + 策略引擎 + 证据链 + Web UI）已经建立，但在**检测性能的执行效率**、**安全边界的闭合程度**和**实时可观测性的完整度**三个维度上存在需要优先修复的缺口。本文档的 P0 建议应在下一轮开发中首先落地，P1 和 P2 按顺序跟进。
