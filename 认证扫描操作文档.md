## 2. 认证配置详解

### 2.1 CLI 参数

| 参数 | 说明 | 示例 |
|------|------|------|
| `--login-endpoint` | 登录页面路径 | `/login.php`, `/api/login` |
| `--login-creds` | 用户名:密码，多个用逗号分隔 | `"admin:password,root:root"` |
| `--login-user-field` | 用户名表单字段名（默认 `username`） | `username`, `email`, `user` |
| `--login-pass-field` | 密码表单字段名（默认 `password`） | `password`, `pass`, `passwd` |

### 2.2 .env 文件配置

在项目根目录编辑 `.env` 文件，添加以下配置：

```env
# ── 认证配置 ──
AUTOPENX_LOGIN_ENDPOINT=/login.php
AUTOPENX_LOGIN_CREDENTIALS=admin:password,test:test123
AUTOPENX_LOGIN_USERNAME_FIELD=username
AUTOPENX_LOGIN_PASSWORD_FIELD=password
AUTOPENX_CSRF_FIELD=user_token

# ── 已有的扫描配置 ──
DEEPSEEK_API_KEY=sk-xxxx
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
AUTOPENX_ALLOW_LOCAL_TARGETS=true
AUTOPENX_EXPLOIT_ENABLED=true
AUTOPENX_MAX_ITER_PER_STATE=6
```

### 2.3 Web UI 配置

启动 Web 界面后，在扫描请求中传入认证参数：

```bash
# 启动 Web UI
python -m autopnex.web.api

```

## 4. 完整命令行参数

```
用法: python autopnex.py [选项]

必需参数:
  --target, -t URL          目标 URL 或主机名

认证参数:
  --login-endpoint PATH     登录页面路径 (如 /login.php)
  --login-creds STR         凭证 user:pass,user:pass
  --login-user-field NAME   用户名字段名 (默认: username)
  --login-pass-field NAME   密码字段名 (默认: password)

扫描控制:
  --mock                    强制离线规则引擎模式
  --max-iter N              每阶段最大迭代次数
  --scan-mode MODE          passive | active
  --allow-external-tools    允许外部工具 (nmap/sqlmap)
  --allow-local-targets     允许本地/内网目标
  --enable-exploit          启用漏洞利用阶段

输出:
  --out, -o PATH            Markdown 报告输出路径
  --html PATH               HTML 报告输出路径
  --json PATH               JSON 原始数据输出路径
  --quiet, -q               减少控制台输出

其他:
  --yes, -y                 跳过授权确认 (CI 模式)
  --list-tools              列出所有可用工具
  --version, -V             显示版本
```

## 5. 扫描流程说明

### 5.1 状态机流程

```
INIT → [自动登录] → RECON → SCAN → VULN_DETECT → EXPLOIT → REPORT → DONE
```

### 5.2 自动登录流程

```
1. 检查 login_endpoint 是否配置
2. GET 登录页面，提取 CSRF token (如 user_token)
3. 依次尝试 credentials 列表中的每组凭证
4. POST 登录表单，检测成功指标:
   - 200 + "logout"/"security" 关键词
   - 302 重定向 + 新 cookie
   - URL 离开登录页
5. 成功后全局 session 自动携带 cookie
6. 后续所有工具的 HTTP 请求自动带上认证 cookie
```

### 5.3 各阶段改进

| 阶段 | 改进内容 |
|------|---------|
| **INIT** | 新增自动登录，全局 session 管理 |
| **RECON** | 端口列表扩展，新增 4280 等容器端口 |
| **SCAN** | 爬虫跟随 302 + 跳过登录页；字典扩展到 99 条 |
| **VULN_DETECT** | 检测工具自动使用认证 session；减少重复调用 |
| **EXPLOIT** | auth_bypass 支持 CSRF token + 已认证检测 |

## 6. 故障排查

### 6.1 登录失败

**症状**: 日志显示 `[login] failed: All login attempts failed`

**排查步骤**:
1. 确认登录路径正确:
   ```bash
   curl -v http://<IP>:<PORT>/login.php
   ```
2. 确认表单字段名（用浏览器 F12 查看）:
   ```
   # 常见字段名
   username / password
   user / pass
   email / password
   log / pwd (WordPress)
   ```
3. 确认凭证正确:
   ```bash
   curl -X POST http://<IP>:<PORT>/login.php \
     -d "username=admin&password=password" -v
   ```
4. 如有 CSRF token，配置 `--csrf-field` 或 `.env` 中的 `AUTOPENX_CSRF_FIELD`

### 6.2 端口未发现

**症状**: `port_scan` 报告 0 个开放端口

**排查步骤**:
1. 确认目标端口:
   ```bash
   nmap -p 1-10000 <IP>
   ```
2. 如果端口不在默认列表中，可在 `.env` 中自定义（暂不支持，需修改代码）
3. 确认网络可达:
   ```bash
   curl http://<IP>:<PORT>
   ```

### 6.3 漏洞未检出

**症状**: 0 True Positive

**排查步骤**:
1. 确认登录成功（查看日志中 `[login] success` 行）
2. 确认爬虫发现了漏洞页面（查看 `discovered_paths`）
3. 确认参数列表非空（查看 `parameters` 数量）
4. 降低安全等级测试:
   ```
   # DVWA 安全等级设为 Low
   # 访问 http://<IP>:<PORT>/security.php 设置
   ```

### 6.4 重复调用过多

**症状**: 同一工具被调用 5+ 次

**说明**: 此问题已在本次修复中缓解:
- `_match_phase_task` 支持模糊匹配
- 连续 3 次拒绝后自动 advance
- 拒绝消息包含 pending tasks 列表

如仍有问题，降低 `max_iter`:
```bash
python autopnex.py --target ... --max-iter 4
```
